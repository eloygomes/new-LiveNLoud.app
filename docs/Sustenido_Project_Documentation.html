1. Executive Summary

This repository contains a multi-service music preparation and live performance platform. The main product identity in the UI is #Sustenido, while several technical and historical artifacts still use the older name LiveNLoud.

The platform combines:

  • A React frontend for dashboard, library management, editing, presentation, tuner, metronome, and profile flows.
  • A Node.js / Express API as the main authenticated application backend.
  • A Python scraping service that imports song content from external music websites.
  • A Python Socket.IO service used in the tuner / footswitch pipeline.
  • A Chrome extension used to capture supported song pages directly from the browser.
  • Docker Compose orchestration for MongoDB, Node, Python scraper, Python Socket.IO, and mailserver services.
Primary User Goal

Manage a personal song library with per-instrument arrangements, embed videos, setlists, lyrics, chords, tabs, and live presentation tooling.

Primary Technical Pattern

Frontend calls Node API. Node delegates scraping to Python when needed. Python stores structured song data in MongoDB and mirrors shared songs into a general catalog.

Core Architectural Rules

  • The React app is the main operator console for authenticated users.
  • Song data is user-scoped in the main app database but also mirrored into a general shared catalog.
  • Scraping is isolated in Python, not in Node, which keeps source-specific parsing logic separate from business routes.
  • The Chrome extension is a lightweight ingestion client and depends on the same backend APIs.
  • Presentation mode is not just read-only display; it includes live performance tooling such as tooltip chords and fullscreen LIVE mode.

4.3 Important Frontend Contexts

Auth Context

Front/src/contexts/AuthContext.jsx is intentionally simple. It stores the access token and user email in localStorage and exposes login() and logout(). Routing protection is currently presence-based, not role-based.

Bluetooth Context / Footswitch Input

Front/src/contexts/BluetoothContext.jsx centralizes BLE and MIDI footswitch integration. It stores action mappings in localStorage, logs events, and dispatches navigation actions such as arrow scroll and page scroll to the current page scroll target.

This context injects a custom data-scroll-REMOVED_MONGO_USER marker to identify the current scroll container. The warning seen in the browser console about data-scroll-REMOVED_MONGO_USER comes from this mechanism and is a known technical debt item.

4.4 Frontend API Layer

The main API client lives in Front/src/Tools/Controllers.jsx. It creates an Axios instance whose base URL is selected in this order:

  1. VITE_API_BASE_URL
  2. http://localhost:3000 when running locally
  3. https://api.live.eloygomes.com as default production fallback

Important frontend controller responsibilities include:

  • User library fetch and per-song fetch.
  • Song creation and exact song updates.
  • Update last-played timestamps per instrument.
  • User profile actions such as username changes and account deletion.
  • Download of raw user JSON data.

4.5 Core Pages

Dashboard

The dashboard is the main operational screen. It supports search, list display, floating action buttons, exports, and version display. It also intentionally disables global page scroll while active.

New Song and Edit Song

These pages are the ingestion and maintenance center of the app. They maintain separate blocks for guitar01, guitar02, bass, keys, drums, and voice.

Recent behavior now documented in the current codebase:

  • Letras links are voice-only sources.
  • If a Letras link is pasted in another instrument, the UI redirects it to Voice and shows a warning.
  • During scraping, the user receives a persistent Carregando... snackbar.
  • After scraping, artist and song values are hydrated back into the form so save actions do not lose metadata.

Presentation

The presentation page is a performance-focused view that loads a song, selects instrument-specific content, and renders it using the custom processSongCifra pipeline. It supports:

  • Chords, tabs, lyrics, and mixed content rendering.
  • Chord tooltips with alternative fingering variation selection.
  • Embedded video toolbox and draggable utility components.
  • Inline cifra editing via TipTap.
  • LIVE fullscreen mode with black background, white lyrics, golden chords, ESC exit, and keyboard scrolling.

The page also contains a fallback path where songLyrics are displayed when songCifra is empty, which is important for Voice/Letras content.

The Chrome extension provides a lightweight “quick add” flow directly from supported source sites. It is designed to reduce friction when importing songs into the shared backend.

  • Chrome Extension/popup.html and popup.css — popup UI.

6.4 Data Persistence Pattern

The Node layer primarily reads and writes user-scoped song documents under the Mongo database liveNloud_. In addition, shared song-like artifacts are mirrored into a separate general catalog used by extension and scraping flows.

6.5 YouTube Integration

The YouTube routes under node/youtube/ implement OAuth URL generation, state verification, token exchange, token refresh, and playlist export. The flow stores YouTube credentials per authenticated user and retries export once automatically when it detects invalid or expired credentials.

7. Python Scraper Service

7.1 Role

The Python service is an isolated HTTP API dedicated to scraping and persisting song content imported from external websites. Its main entrypoint is Server/liveNloud/python/scrapper.py.

7.2 Scraper Request Flow

  1. Receive a POST request at /scrape.
  2. Sanitize and normalize the incoming link.
  3. Detect the source host using source_rules.py.
  4. Reject invalid source/instrument combinations such as Letras on non-Voice fields.
  5. Dispatch to the correct source-specific scraper through scraping_service.py.
  6. Persist the normalized result to MongoDB through storage_service.py.
  7. Return songData so the frontend can hydrate its form immediately.

7.3 Supported Scraping Sources

Source Python Service Notes
Cifra Club scraping_service_cifraclub.py Original/core source.
Ultimate Guitar scraping_service_ultimate_guitar.py Uses UG API first, keeps browser scraping fallback.
Letras / Letras.mus.br scraping_service_letrasmus.py Lyrics-oriented source intended for Voice instrument.

7.4 Important Current Scraper Rules

  • Letras links are Voice-only. If the request uses another instrument, the API returns a 400 with a guidance message.
  • Lyrics-only sources can omit capo/tuning/tab fields. Storage logic now handles missing values safely.
  • Returned payload includes songData. This is required by the frontend and extension to hydrate title/artist immediately after scrape.

7.5 Storage Service Responsibilities

storage_service.py turns scraper output into the internal song schema. It creates a per-instrument block containing fields such as:

  • active
  • capo
  • tuning
  • lastPlay
  • songCifra
  • songTabs
  • songChords
  • songLyrics
  • progress
  • link

It also forwards normalized data to the general catalog through the Node API after Mongo persistence.

7.6 General Catalog Mirroring

The Python service does not only persist into the user-scoped library. It also sends a normalized version to the shared generalCifras path so that the platform can resolve songs by instrument, link, artist, and title across flows such as extension add and scraper retries.

8. MongoDB Data Model

8.1 User-Scoped Song Model

The dominant structure in the application stores one document per user email with a userdata array of songs. 

{
  "email": "user@example.com",
  "userdata": [
    {
      "id": 1,
      "song": "Wish You Were Here",
      "artist": "Pink Floyd",
      "capo": "",
      "tom": "",
      "tuning": "",
      "progressBar": 0,
      "instruments": {
        "guitar01": true,
        "guitar02": false,
        "bass": false,
        "keys": false,
        "drums": false,
        "voice": true
      },
      "embedVideos": [],
      "setlist": [],
      "addedIn": "YYYY-MM-DD",
      "updateIn": "YYYY-MM-DD",
      "guitar01": { ...instrument block... },
      "voice": { ...instrument block... }
    }
  ]
}

8.2 Instrument Block Design

Each instrument stores its own content independently. This is one of the key architectural choices in the application: the same song can hold multiple arrangements and source links at once.

8.3 General Catalog Model

The shared catalog in generalCifras stores song-level documents that can be resolved by source link and by instrument availability. This is especially important for extension-based ingestion and for scraper-to-frontend synchronization.

11. Current Strengths of the Codebase

  • Clear separation between scraping logic and the main Node business API.
  • Per-instrument storage model is flexible and well aligned with rehearsal/performance use cases.
  • Chrome extension significantly reduces friction for content ingestion.
  • Presentation page now supports true stage-focused fullscreen LIVE mode.
  • Ultimate Guitar and Letras integration is already wired end-to-end across backend, frontend, and extension.

12. Important Technical Debt / Risks

  • Very large backend file: Server/liveNloud/node/index.js mixes active code and large commented historical blocks, which increases maintenance risk.
  • Mixed naming: product branding is moving from LiveNLoud to #Sustenido, but many internal identifiers still use the old name.
  • LocalStorage-heavy state: several frontend flows rely on localStorage as operational state, not just session state.
  • Console warning in Bluetooth context: custom DOM attribute injection currently produces React warnings.
  • Legacy folders remain in repo: Server/OLD and built deployment artifacts can confuse new maintainers.
  • Large monolithic controllers/components: some frontend pages and the main controller layer are difficult to reason about because of broad responsibilities.

13. Recommended Refactoring Priorities

  1. Split Server/liveNloud/node/index.js into route modules by domain: auth, songs, profile, scraping, YouTube, media.
  2. Document the Mongo schema formally in a dedicated versioned spec.
  3. Replace ad-hoc frontend localStorage field passing with a normalized state strategy.
  4. Remove or archive obsolete deployment artifacts that are no longer part of the active system.
  5. Centralize source definitions and field contracts for extension, frontend, and Python scraper to reduce drift.
  6. Add explicit tests around scraper normalization and instrument persistence, especially for lyrics-only sources.

14. Operational Workflows

14.1 Add Song Through Frontend

  1. User opens New Song.
  2. User pastes source link into an instrument field.
  3. Node forwards scrape request to Python.
  4. Python detects source and scrapes content.
  5. Python stores normalized result and returns songData.
  6. Frontend hydrates song title, artist, and instrument content.
  7. User saves final record in the app library.

14.2 Add Song Through Chrome Extension

  1. User opens a supported source website.
  2. Content script extracts page metadata.
  3. User opens popup and reviews source, instrument, and metadata.
  4. Popup calls backend save flow.
  5. Backend scraper/general catalog persistence completes.
  6. The song becomes available in the application library and presentation flow.

14.3 Live Performance Flow

  1. User opens a presentation route for a chosen song and instrument.
  2. Frontend loads the exact song entry from the backend.
  3. Rendered content is normalized through the cifra parser.
  4. User can show videos, edit the cifra, or switch content type.
  5. User can enter LIVE mode for fullscreen stage display.
  6. Keyboard arrows, page keys, BLE, or MIDI mapping can be used to navigate the content.

16. Conclusion

The current repository is not a simple CRUD app. It is a hybrid music operations platform that combines song ingestion, per-instrument arrangement storage, external-source scraping, live presentation tooling, browser extension capture, and export/utility integrations. The most important architectural idea in the system is that one song can be represented differently per instrument and still remain part of a unified library and performance workflow.

The codebase is already capable of supporting a full rehearsal-to-stage flow. The next major engineering gains will come from modularization, formal schema documentation, and reducing the current amount of duplicated or historical code retained in the active repository.
