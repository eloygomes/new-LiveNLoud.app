### 2.2 Frontend: persistência, offline e sincronização

A camada de controllers ganhou uma base mais robusta para lidar com modo offline e sincronização de alterações.

#### Arquivos impactados

- [Front/src/Tools/Controllers.jsx](Front/src/Tools/Controllers.jsx)

#### O que mudou

- maior suporte a **offline mode**;
- cache e merge de dados locais de músicas;
- backfill de detalhes de músicas offline;
- fila de mutações pendentes para sincronização;
- tratamento de atualização de song data, last play, notas e upload de Guitar Pro;
- melhorias no fluxo de fallback quando a API não responde.

Isso indica uma tentativa clara de deixar a experiência mais resiliente, especialmente para uso sem conexão.

## 3. Impacto funcional observado

### 3.1 O que ficou mais forte

- **Experiência de apresentação mais completa**
  - suporte a transposição;
  - edição dentro da própria página;
  - controles visuais mais ricos;
  - melhor adaptação mobile/touch.

- **Resiliência offline melhorada**
  - cache local;
  - fila de sincronização;
  - recuperação de song data mesmo sem API.

- **Armazenamento mais consistente para layout da apresentação**
  - a camada server passou a apoiar layouts salvos por instrumento.

### 3.2 O que merece atenção

- O conjunto de mudanças em [Front/src/Pages/Presentation/Presentation.jsx](Front/src/Pages/Presentation/Presentation.jsx) é grande e centraliza muita lógica. Isso aumenta o risco de regressões em cenários que envolvem:
  - live mode;
  - touch layout;
  - edição de cifra;
  - sincronização entre estado local e dados do backend.

- A integração entre `Presentation.jsx`, `ToolBox.jsx`, `TollBoxAcoord.jsx` e [Front/src/Tools/Controllers.jsx](Front/src/Tools/Controllers.jsx) ficou mais complexa, então qualquer mudança futura aqui deve ser validada com foco em fluxo real.

- O servidor também passou a lidar com mais estruturas de dados, o que aumenta a importância de validar compatibilidade com registros antigos já existentes no banco.
